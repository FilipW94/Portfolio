Layout and Mock-ups

Because Dino Dynasty Evolved was meant to be played with a controller, the general game menu layout i opted for was:

  • Menu buttons at the top of the screen which are cycled through using controller shoulder buttons. (LB/RB)
  • Some windows have sub tabs, namely settings and skill tree. These are navigated through using the shoulder triggers. (LT/RT)
  • Elements within each tab are navigated between using left stick or D-PAD.
  • Elements are interacted with either by navigating to and then pressing the A button. Or by direct bindings done through the CommonUI plugin. When using direct bindings the button will respond to the bound input as long as the parent widget is focused and activated.

Popup System




The Popup system was made to be reusable everywhere and has 2 available buttons. The default button that is always visible is Confirm. The second optional button is a Cancel button. The popup widget also has an OnPopupReply event that returns a boolean. This can be used to run different logic depending on the return value.

General uses for the Popup is to display information that you need to make sure the player doesn't miss, hence why its removal is dependent on input. A specific example of when both the Confirm and Cancel button is used is when the player is assigning keybindings via the control settings. If they try to bind a button that already has an active binding, the Confirm button will swap the input bindings and save, while Cancel will prevent the action. Both inputs then close the popup.

The Popup widget can also display an Image or Video. This can further clarify new mechanics or systems that simple text might struggle to convey. In DDE we opted to use short ability previews on the popup whenever a player unlocked a new base ability. As well as making them viewable from their corresponding skill tree by pressing Y. This helped the player grasp the ability quicker than just reading.

Below you have a typical setup when using the popup system. You create the widget and call an interface function on the HUD class attached to the player controller. This will safely open the widget above any other UI elements and set focus on the widget.

Prompt System

The prompt system, similar to the Popup system was made to be reusable everywhere. It implemented an interface that used 4 different functions. (click any image to view enlarged)

  • PromptId: Used to identify prompts and allows for several to exist at once. As well as controlled removal.
  • VisibleDuration: Will show the prompt for this amount of time before removing it. Setting this value to 0 will leave it up until ClearSpecificPrompt or ClearAllPrompts is called. Alternatively calling UpdateActivePrompt with a new duration.
  • PromptTitle: Title for prompt, just a word or two. If this is left empty the text component holding the title text will be collapsed. Allowing the PromptInfo to use up max amount of space.
  • PromptInfo More in-depth description of what information the prompt is meant to relay to the player. If this field is left empty. The text component holding the info text will be collapsed and the PromptTitle text component will be centered on the prompt window.

  • CreateNewAsBackup: This variable in UpdateActivePrompt was a minor safety implementation that would just create a new prompt if the one you are looking for is no longer active. Mainly because many different people would use the system and this would make sure the prompt was added no matter what.

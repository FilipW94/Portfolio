Description

Dino Dynasty Evolved is a cinematic third-person action-adventure set in a prehistoric world. Lead your herd through the Cretaceous era as you explore diverse biomes, battle rival species, and unlock new abilities to survive and dominate.

Check out Dino Dynasty Evolved on Kickstarter!

My time

Dino Dynasty Evolved is a game I worked on for 8 months between May-December 2025. Which was expected to be about half of the production time. I was there in an intern capacity, and initially I was meant to work as system/tech designer. But since I had been thinking of switching to UI/UX design with focus on implementation and the more technical side, I reached out to our contact person at the company and asked to be in charge of UI for the project. To which they agreed.


Inter-project collaboration

During my time at ION I worked very closely with several other departments to accomplish my tasks, including:

  • Other members of the design discipline with different specializations.

Layout and Mock-ups

Because Dino Dynasty Evolved was meant to be played with a controller, the general game menu layout I opted for was:

  • Menu buttons at the top of the screen which are cycled through using controller shoulder buttons. (LB/RB)
  • Some windows have sub tabs, namely settings and skill tree. These are cycled through using the shoulder triggers. (LT/RT)
  • Elements within each tab are navigated between using left stick or D-PAD.
  • Elements are interacted with either by navigating to and then pressing the A button. Or by direct bindings done through the CommonUI plugin. When using direct bindings the button will respond to the bound input as long as the parent widget is focused and activated.

Popup System




The Popup system was made to be reusable everywhere and has 2 available buttons. The default button that is always visible is Confirm. The second optional button is a Cancel button. The popup widget also has an OnPopupReply event that returns a boolean. This can be used to run different logic depending on the return value.

General uses for the Popup is to display information that you need to make sure the player doesn't miss, hence why its removal is dependent on input. A specific example of when both the Confirm and Cancel button is used is when the player is assigning keybindings via the control settings. If they try to bind a button that already has an active binding, the Confirm button will swap the input bindings and save, while Cancel will prevent the action. Both inputs then close the popup.

The Popup widget can also display an Image or Video. This can further clarify new mechanics or systems that simple text might struggle to convey. In DDE we opted to use short ability previews on the popup whenever a player unlocked a new base ability. As well as making them viewable from their corresponding skill tree by pressing Y. This helped the player grasp the ability quicker than just reading.

Below you have a typical setup when using the popup system. You create the widget and call an interface function on the HUD class attached to the player controller. This will safely open the widget above any other UI elements and set focus on the widget.

Prompt System

The prompt system, similar to the Popup system was made to be reusable everywhere. It implemented an interface that used 4 different functions.

  • PromptId: Used to identify prompts and allows for several to exist at once. As well as controlled removal.
  • VisibleDuration: Will show the prompt for this amount of time before removing it. Setting this value to 0 will leave it up until ClearSpecificPrompt or ClearAllPrompts is called. Alternatively calling UpdateActivePrompt with a new duration.
  • PromptTitle: Title for prompt, just a word or two. If this is left empty the text component holding the title text will be collapsed. Allowing the PromptInfo to use up max amount of space.
  • PromptInfo More in-depth description of what information the prompt is meant to relay to the player. If this field is left empty. The text component holding the info text will be collapsed and the PromptTitle text component will be centered on the prompt window.

  • CreateNewAsBackup: This variable in UpdateActivePrompt was a minor safety implementation that would just create a new prompt if the one you are looking for is no longer active. Mainly because many different people would use the system and this would make sure the prompt was added no matter what.

Input Icons

Since we used both KBM and Controller, as well as a key-rebinding system. We needed a way to display what input each action was bound to. For that reason I set up a:

  • Material to show the icons
  • Data Table to hold all the keybinding icons
  • Functionality to fetch and display the icon

M_IconPicker

The material used to display the icons uses a sprite sheet with the icons for each device.


The user assigns the grid dimensions for the sprite sheet, the row and column indexes are then used to find the correct icon.

A material instance was created for each input device.

  • MI_IconPicker_KBM
  • MI_IconPicker_Xbox
  • MI_IconPicker_Playstation

DT_ButtonIcons

The data table uses the row name of the actual input. The row then contains the material instance of the input device, row index and column index. All things you need to display the icon.

This setup reduced the need for 100+ material instances down to 3, while also reducing the amount of time it takes to assign each row since you don't have to locate the asset everytime. You just copy the row and type the buttons row/column index.

Retrieving Row Data

Now that a collection of all the input icons had been created, I just needed to fetch it. This was done by having a reference to the input action tied to an ability, the gameplay input mapping context and discerning what input device is being used. With this information I could find the input name and use it to fetch the input icon data from the data table.

This logic was then ran when:

  • The game started
  • When an action was re-bound
  • when the event OnInputMethodChanged fired (CommonUI plugin event)

In the video I use xbox controller and re-bind the bite ability from Y to B. Then I also start using my keyboard and mouse which triggers the OnInputMethodChanged event.

Acquisition Log

The game had 3 resources, and we needed to display acquisition in a clear but non-intrusive way.


When a resource is acquired it will show the corresponding resource icon with the amount gained. If no new resources is gained within 1.5 seconds it will fade out. If a new resource is collected it will update the acquisition log again and reset the duration. All resources always fade-out simultaneously

If a resource is collected while the fade out widget animation is playing, it will reverse the fade-out, making it look smooth.

Originally we had the total resources displayed center-left instead of bottom-left. During this time I also made an animation where the gained amount shot into the total amount. But since collecting resources is something the player does frequently, we opted for a less visually stimulating effect.

Ability Button Visuals

The ability button is an indicator for the different characteristics of an ability. This includes key-binding, cooldown, held duration and charges.

To display all this information I made use of the internal space for the cooldown and held-progress indication, while showing the keybinding and available charges on the outside of the indicator for abilities that had charges.

(Cooldown text and ability name were not meant to be in the final game, only for development)

Since the ability buttons are placed in an arc in the bottom-left corner, their rotation had to be manipulated. This caused the ability icon (white) and keybinding to be rotated as well. this was solved by creating a function in the ability button class that checks its own rotation, and then subtracts that from the keybinding and icon.

At the time of implementation the amount of max charges an ability could have hadn't been decided yet. So I made it work with up to 4 charges by creating simple textures that are assigned in the child blueprint class. The ability button then checks how many charges it's corresponding ability has and applies the correct one.


Charges are then displayed by checking CurrentCharges/MaxCharges. At the time my internship we had plans to add new textures for charges but were using the old ones from when our ability button was circular. That's why they don't fit very well.

UI Navigation

For The general horizontal tab layout I used the CommonUI Tablist class. It has functionality that makes the input bindings easy to set up and also keeps it separate from the other gameplay inputs.


Our settings menus use a settings slider class placed inside either a vertical box or a scroll box. A problem here was that we wanted to use vertical analog inputs for navigation, but horizontal analog inputs to change the settings option. For this reason I didn't use the default navigation system and instead set up some custom logic.

The parent widget holding all the sliders consume the vertical analog inputs and move between the different slider.

The settings sliders themselves consume the horizontal analog inputs and change the current settings option.

One of the trickier parts of the navigation was the skill tree. Since we had a large amounts of skills, a simple vertical/horizontal layout didn't work. Both because the sheer amount wouldn't fit in the large skill trees, but also because of the connections between them. To set this up I used the default navigation which uses vertical/horizontal movement and tested moving between all the nodes to find what felt good or not.

UI Materials

Skill Tree Node Material

This material was used for the skill tree nodes. It needed functionality to display several different states like:

  • Dormant: Default state for skill tree node. Available further down the skill tree unless a mutually exclusive skill is unlocked.
  • Available: The player can unlock this if they have the required resources and are inside a friendly nest.
  • Unlocked: The player has already unlocked this skill.
  • Locked: The player has unlocked a mutually exclusive skill.


Ability Button Material

Used for the ability indicator. Is split up into background and icon with both having gradual color changes. The background is used to indicate cooldown duration and the foreground icon is used to show held progress.

Health Bar Material

Used to display health in an arcing health bar. Basically just a circular progressbar but the anchor is set to the bottom left corner with the rest hidden.

Controller Feedback

During my time working on DDE I didn't have too much time to work with controller feedback. I did however set it up for a couple of our menu elements that required the player to hold down a button for a short while. Mainly while unlocking a new tier and abilities

I had already made states for both these components to make it easier to branch logic.

In the example above the controller feedback is 2 seconds long. Same as the required hold duration. If the player releases the button or it finishes. The feedback ends.

An added bonus was to also check the current input type. This would prevent a plugged-in controller to start vibrating when playing with mouse and keyboard, which can be an annoyance.
